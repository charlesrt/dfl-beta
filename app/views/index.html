Design prototypes for

Search relevance and journey

Different prototypes have been created to test specific scenarios and interactions for how we might implement solutions that work in the future.

Before you start:

  • This is not a real product or service. You should only continue if you have been invited to test this prototype.
  • This prototype is intended to communicate and test functional ideas only. All visual design is illustrative only for prototyping purposes and not to be used for comparison to delivered product(s).
  • Data entered is stored locally only for the duration of your session.

Prototypes

[Latest] Search journey (Sprint 4) - Iterated version of dataset view for delivery. Removing breadcrumbs in favour of back link. Removing download all and copy links. Adding follow action and reorganising resource content as table layout surfacing most important information for comparison and scanning.

Search journey (Sprint 3) - Design candidates for the dataset view (the content item selected as part of search) surfacing detailed dataset data making a futher sub-level view redundant. Including total file data and download all option. Also considering breadcrumbs for user orientation and abiity to return to search.

[Latest] Search relevance (Sprint 2) - Bringing the components together from past experimentations to create a complete view of intent for improving search relevance through, improved meta data, string highlighting, filtering interactions and more.

Metadata variants in search results (Sprint 2) — Exploring multiple options how metadata might be surfaced in search results. These were created to visualise different ideas and approaches so that a decision could be made balancing desirability and feasibility.

Filter by types of data (Sprint 2) — Mapping individual file types/extensions to collective data types, for example report or table (rather than .pdf or .xls). This is concept to be explored as an enhancement to the product in the future.

Previous experiments (Sprint 1) — Design experiments to test the highest priority assumptions. Using the evidence from research the team will be able to make new decisions on priority, progress certain tickets to development and identify areas for iteration.
